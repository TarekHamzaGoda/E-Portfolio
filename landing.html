Forum Discussion: Factors that Influence Reusability

One important aspect of Object-Orientated Programming is code reusability. Over the years, software engineers were able to advance in OOP practices across all industries by leveraging existent codes. OOM or object-oriented metrics aims to measure the reusability of software codes and data organization that can assist with efficient OOP development. In an article by Padhy et al (2018), researchers have established several properties in which code quality is measured.

  1. “Architecture driven approach (ADP)”
  2. “An Algorithm used in the program (AP)”
  3. “Modules in the program (MIP)”
  4. “Requirement Analysis (RA)”
  5. “Design Patterns (DP)”
  6. “Documentation in project (DIP)”
  7. “Test Cases/Test Design (TCTD)”
  8. “Used in the data project (UD)”
  9. “Models in the project (MP)”
  10. “Knowledge requirement (KR)”
  11. “Service contracts (SC)”

List priority above is organized based on my personal opinion.

architecture ADP and algorithm AP are the main structure of the system. Code reusability often depends on the success of both aspects. Thus, ADP and AP are one of the first indicator of code quality. MIP, RA, and DP are tested with respect to project requirements. Similarities in implementation can also indicate the reusability of the code. Another important code quality indictor is TCTD, which in return insure the validity of the pervious reusable properties.

UML models, Software Development Life Cycle

Unified modelling language or UML is a modelling technique the illustrates a detailed development process of a system (Jacobson et al. 2004). In other words, UML is a general-purposed language that tends to support a step-by-step process to achieve a practical system. UML practices is considered a good development process and is crucial to the success of a software. Therefore, there are several types of UML models thar are used in different stages of the development process.

  1. User Requirements Stage: Use Case Diagram, Activity Diagrams, Deployment Diagram
  2. System Implementation Stage: Class Diagram, State Machine Diagram
  3. Development Stage: Use Case Diagram, Sequence Diagram, State Machine Diagram
  4. Validation and Testing Stage: State Transition Diagram
  5. Packaging and publishing Stage: Activity Diagram.

Data validation is the process of ensuring that the data type input is correct. At first, I found it difficult to make the connection between Encapsulation and Data validation. However, after practicing the below code, the syntax stared to make since. 


@property
def name(self):
	return self._name
@name.setter
def name(self, new_name):
	if type(new_name) != str:
	raise TypeError("Names must be expressed as a string")
	self._name = new_name

The above syntax represents 2 class methods with a property to decorator to restrict access to the attribute and setter to change the value. Validating in this case, is performed through an exception built-in function in python. This will ensure the correct data being inputted thus the data is validated.

Unit 8

Forum Discussion: Metamodel Design to support OOD of the IoT

Object-Oriented Programming (OOP) is a foundational aspect of software development. Therefore, Fortino et al. (2015) introduces a new approach that aims to apply similar fundamentals to the Internet of Things (IoT). OOP initiatives require planning and design during early development stages. systematic software design strategies such as UML, helps provide a blueprint to support the development. Similarly, Metamodel is an approach that supports analysis and implementation techniques that aids with the development of Smart objects to the IoT (Fortino, et al., 2015). Metamodel approaches can strongly support and identify smart object functions in different abstraction levels. Early development phases must require efforts to understand main system design, relationships, and functional components. As such, metamodel approaches can provide abstract testing grounds for features in different phases of the system. Although system design planning approaches such as UML or Metamodel are supplemental factors of system development, it remains a challenge to include actual syntax within both approaches. That is because code components, functions, behaviours and algorithms are subject for constant change due to numerous factors during system implementation (Perez & Sellers 2008). A metamodel in concept provides a generalized approach of the system design. The inclusion of further implementation strategies can be time consuming and maybe inefficient.

Cyclomatic Complexity

CC or cyclomatic complexity is unit that defines the possible number of paths inside a code and calculated using McCabe’s Formula:

Complexity = Graph edges – Graph Nodes + 2(Connected Components)


M = E – N + 2P
									

									
A Code can be translated into graphs in order to calculate its complexity. This exercise will be applied on the following code.
